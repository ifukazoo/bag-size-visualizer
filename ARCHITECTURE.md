# アーキテクチャ

Bag Size Visualizer の構成と設計判断をまとめる。コードに埋め込まれた意図を俯瞰できるようにすることが目的。

## 1. 技術スタック

- React 19 + TypeScript / Vite / Tailwind CSS v4
- [`@imgly/background-removal`](https://github.com/imgly/background-removal-js)（WASM / ONNX によるブラウザ内背景除去）
- テスト: Vitest（単体テスト）/ Playwright（E2E）
- CI: GitHub Actions
- ホスティング: Cloudflare Workers（Static Assets）

## 2. 全体構成

```
┌─────────────────────────── ブラウザ（クライアント） ───────────────────────────┐
│                                                                                │
│  React SPA                                                                      │
│   ├─ 画像アップロード                                                            │
│   ├─ 背景透過  … @imgly/background-removal（WASM / ONNX をブラウザ内実行）        │
│   ├─ 合成・実寸換算  … <canvas> へ 3 層描画                                       │
│   └─ PNG 書き出し  … OffscreenCanvas → Blob → ダウンロード                       │
│                                                                                │
└────────────────────────────────────────────────────────────────────────────┘
                              ▲ 静的アセット（HTML/JS/CSS/画像/WASM）
                              │ HTTPS（GET のみ）
                  ┌───────────┴───────────┐
                  │ Cloudflare Workers     │  ← 静的アセット配信のみ。サーバーロジックなし
                  │ (Static Assets)        │
                  └────────────────────────┘
```

バックエンド API・データベース・認証は持たない。Worker は静的アセットを返すだけで、サーバーサイドのコードは無い。

## 3. 設計判断と根拠

### 3.1 なぜ完全クライアントサイドか

| 観点 | 理由 |
|---|---|
| プライバシー | アップロード画像をサーバーに送らない。利用者の写真が外部に出ない |
| コスト | サーバー計算が無い。重い背景透過は来訪者の端末で実行されるため、アクセスが増えても処理コストが増えない |
| スケール | 静的配信のみなので、トラフィックが跳ねても破綻しない |

### 3.2 なぜ Cloudflare Workers（Static Assets）か

- **静的アセットの配信は無料・無制限**で、アクセス集中でも従量課金が発生しない。
- Cloudflare の **現行推奨パス**（旧来の Pages ではなく Workers + Static Assets）に沿う。
- サーバーロジックが無いため `main`（Worker スクリプト）すら不要で、`wrangler.jsonc` の `assets.directory` に `dist/` を指すだけで配信できる。
- SPA のため未知パスは `index.html` へフォールバックする（`not_found_handling: single-page-application`）。

### 3.3 背景透過をブラウザ内で実行する

背景透過は [`@imgly/background-removal`](https://github.com/imgly/background-removal-js) を使い、WASM / ONNX ランタイムを **来訪者のブラウザ内** で動かす。

- **初回は推論モデル（約 50MB）の取得が走る**ため時間がかかる。UI に進捗を表示する。
- マルチスレッド WASM は `SharedArrayBuffer`（= `crossOriginIsolated`）を要求する。現状は COOP/COEP ヘッダを付けておらず、**単一スレッドにフォールバック**して動作する。
  - トレードオフ: 追加ヘッダ無しで確実に動くことを優先した。高速化したい場合は COOP/COEP を付与してマルチスレッド化できるが、その場合はクロスオリジンのモデル取得や自己ホストの調整が必要になる（→ 7. 既知の制約）。

### 3.4 自動透過の限界と「二段構えの入口」

自動背景透過は万能ではない。**被写体と背景のコントラストが低い**（例: 淡い色のバッグを明るい背景で撮影）と、バッグ本体まで背景と判定されて欠けることがある。

そのため画像の入口を 2 つ用意している:

| 入口 | 用途 |
|---|---|
| 自動で背景透過して使う | 背景つき写真をその場で透過 |
| 透過済み画像をそのまま使う | 別途用意した透過 PNG を無加工で使う（自動透過が苦手なケースの逃げ道） |

## 4. 描画パイプライン

合成は `<canvas>` に対して次の 3 層を重ねる（`src/hooks/useBagCanvas.ts`）。

```
① シルエット（下地）
② バッグ画像（②は①と③の間）
③ 手レイヤー（手だけの透過 PNG・最前面）
```

③を最前面に置くことで「手でバッグを持っている」見え方になる。

### 4.1 不透明 bbox トリミング

背景透過後の画像は「元写真と同寸・背景だけ透明」なので、そのまま枠に割り当てると透明な余白の分だけバッグが小さく描かれてしまう。
そこで不透明ピクセル（α>16）の **最小包含矩形** を求め、その矩形だけを実寸枠に割り当てる（`opaqueBboxFromImageData`）。不透明ピクセルが無ければ画像全体にフォールバックする。

### 4.2 実寸換算（頭足ピクセル基準）

サイズ感の正しさはこのツールの核。身長に対応するのは**画像全高ではなく「頭頂〜足先」のピクセル**である点が肝になる。

- シルエット画像中の頭頂 Y・足先 Y から `BODY_RATIO`（頭足ピクセル / 画像全高 ≒ 0.94）を定義。
- 描画されたシルエット高さ × `BODY_RATIO` ÷ モデル身長(cm) = `pixelsPerCm`。
- バッグの cm 指定に `pixelsPerCm` を掛けて描画サイズを決める。

画像全高を身長として扱うと `pixelsPerCm` が過大になり、バッグがわずかに大きく描かれてしまう。これを避けるための補正。

### 4.3 WYSIWYG（プレビューと出力の一致）

プレビューは 800px、書き出しは 1080px。両者で見た目を完全に一致させるため、レイアウト計算（`calcLayout`）を **`canvasSize` に対して一様にスケール** するよう純関数化している。
キャプション帯も `CAPTION_BAND_RATIO`（= 0.1）という比率で確保し、プレビュー／出力で同じ比率を使う。これによりシルエット配置が厳密な相似になり、**ドラッグ位置のマッピングがズレない**。

### 4.4 ドラッグ（再描画を避ける）

バッグ位置・ドラッグ状態は React state ではなく **ref** に持つ。ドラッグ中は `draw()` を直接呼んで `<canvas>` を更新し、React の再レンダリングを発生させない（毎フレームの再レンダリングを避けるため）。

## 5. ロジック分離とテスト戦略

副作用（DOM・Canvas）と純粋計算を分離している。

- **純粋計算** … `src/lib/bagLayout.ts`（`opaqueBboxFromImageData` / `calcLayout` / `resolveBagPos`）。DOM に依存しないため **vitest で単体テスト**（実寸換算・WYSIWYG の一様拡大・bbox 境界値など）。
- **DOM/Canvas/フック** … `useBagCanvas.ts` は上記純関数を呼び出す薄い層に保つ。
- **エンドツーエンド** … Playwright でアップロード→操作可能化→PNG ダウンロードの一連を検証（`e2e/`）。
  - 自動背景透過は大容量モデル取得を伴い CI で不安定になりやすいため、E2E からは除外し、計算ロジックは単体テストで担保する。

## 7. 既知の制約・将来の改善

- **背景透過の高速化**: COOP/COEP ヘッダを付与して `crossOriginIsolated` を有効化すればマルチスレッド WASM が使える。ただしクロスオリジンのモデル取得との両立が必要。
- **モデルの自己ホスト**: 推論モデルを同一オリジンから配信すれば初回取得が安定するが、**Workers Static Assets の 1 ファイル上限 25MiB** に注意（ONNX ランタイム wasm は現状でも約 23MB と上限に近い）。
- **自動透過の精度**: 低コントラスト被写体での欠けは「透過済み入口」で回避する設計だが、前処理（コントラスト補正等）での改善余地はある。
